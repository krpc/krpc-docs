# Client library API parity audit

**Status:** audit (2026-07-21) — fixed: bugs 1-8 (cnano and Java), 9, 12-13 (C++), 14-15 (C#),
17-19 (Lua), 20-22 and part of 23-48 (Python), and 49-51 — see the "bugs, as fixed" sections.
Outstanding: 10-11 (C++ teardown), 16 (C#), and the rest of the Python sweep. Every client has
now been swept at least once; only Python has had a dedicated one. Bugs 23-48 come from a dedicated sweep of the Python client, run after
bugs 20-22 showed that treating it as the reference had left it the least examined client here;
see "Python client sweep" for the findings and for what that says about this audit's method.

An audit of the six bundled client libraries (Python, C++, C#, Java, Lua, cnano) covering
three questions: which client-runtime APIs each one implements, whether they take the same
approach so concepts transfer between languages, and which server/protocol features no
client exposes.

Scope is the **hand-written client runtime** — connection, streams, events, encoding,
expressions — not the per-service stubs generated by `tools/krpctools/clientgen`. Python is
treated as the reference implementation throughout.

Several findings here already have agreed designs and issues; those are cross-linked in
"Already tracked" rather than re-proposed. The new material is the parity matrix, the
non-stream gaps (timeouts, reconnection, error fidelity, Lua/cnano scope), and the bug list.

## Where the clients already agree

These conventions are consistent across every stream-capable client and are worth
preserving as the house style:

- Streams are created stopped (`AddStream(call, false)`) and auto-start on first read.
- Deduplication is keyed on the server-returned stream id, not on a client-side call cache.
- A stream error is stored as the stream's value and rethrown at read time.
- Waiting uses a condition variable that the **caller must lock first**; this is documented
  per client and not enforced in code.
- Callbacks fire on the stream update thread.
- One call per request; a single lock serializes each RPC round trip.
- Expression factories take the connection as an explicit first argument.

## Divergences, ranked

**1. Stream creation has five different mental models.** Nothing transfers between clients:

| Client | Syntax |
|---|---|
| Python | bound method plus args — `conn.add_stream(f, a)`, `getattr` form for properties |
| C++ | generated `_stream`-suffixed methods — `vessel.position_stream(rf)` |
| C# | LINQ lambda — `conn.AddStream(() => vessel.Position(rf))` |
| Java | reflection on a **string** method name — `conn.addStream(v, "getPosition", rf)` |
| Lua, cnano | not supported |

Each is idiomatic in isolation, and unifying them fully is neither possible nor desirable.
But the Java string-name form is the weakest of the four: it is not refactor-safe and not
compile-time checked. Its lookup was also broken outright (bug 5, since fixed), which is what
a string-keyed API costs — none of the other three can misresolve, because the compiler does
the resolving. If any one moves toward the others, it should be Java.

**2. Lua and cnano are RPC-only.** Both are documented as such
(`doc/src/lua/client.rst:59-62`, `doc/src/cnano/client.rst:194-200`). For cnano this is a
deliberate embedded-target constraint — streams need a second concurrent connection and
asynchronous receive, and `src/krpc.options` caps messages at one call/one result.

For Lua the gap is sharper than the docs imply. Because services are built dynamically from
`GetServices`, the KRPC service procedures `add_stream` / `start_stream` / `set_stream_rate`
/ `remove_stream` / `add_event` **do appear** on `conn.krpc` as callable stubs with no
runtime behind them. `add_stream` is in practice uncallable anyway, because Lua is the only
client with no way to build an un-executed `ProcedureCall`. The structural blocker is the
concurrency model: one blocking socket, no scheduler, and `platform.sleep` is a busy-wait
loop, so a stream update reader has nowhere to live.

**3. Java cannot make an RPC-only connection.** The stream socket is opened unconditionally
in the constructor (`Connection.java:158`). Every other client can skip it.

**4. C++ discards server stack traces.** `schema::Error.stack_trace` is populated by
the server when `VerboseErrors` is on, and Python, C#, Java and Lua all append it to the
exception message. C++ uses only `error.description()`.

cnano was worse — it decoded the whole `Error` message and threw away service, name,
description and stack trace, collapsing every server-side failure to `KRPC_ERROR_RPC_FAILED`,
so a cnano user could not distinguish any two server errors. This is now fixed as bug 3 below,
opt-in behind `KRPC_ERROR_MESSAGES` because surfacing strings is not free on the targets cnano
is built for. C++ remains outstanding.

**5. Scoped cleanup exists only in Python** (`with conn.stream(...)`). C++ is the worst case:
`StreamManager` holds only `weak_ptr<StreamImpl>`, so when the last `Stream<T>` handle dies
the server-side stream is never removed and leaks for the life of the connection. C# and Java
simply do not implement `IDisposable`/`AutoCloseable` on their stream types.

**6. No client has any timeout or reconnection support.** No client sets a connect timeout, a
socket read/write timeout, or an RPC deadline; a hung server hangs the caller forever. The
only timeouts anywhere are on `Stream.wait` / `Event.wait`. No client can reconnect — the
user must construct a new connection. For cnano this is the highest-value gap of all, since
`VTIME` on POSIX and `Serial.setTimeout` on Arduino are nearly free and their absence lets a
server hiccup hang a microcontroller permanently. (Arduino now has a read timeout, as part of
bug 2; POSIX and Windows still have none, and no client has a connect or RPC timeout.)

**7. Nobody has expression sugar.** Every client writes fully explicit nested factory calls:

```
Expression.GreaterThan(conn, Expression.Call(conn, call), Expression.ConstantDouble(conn, 1000))
```

Each node also costs its own round trip. C++ has a real constraint here — `krpc::Object<T>`
already overloads `==` for identity comparison, so overloading it on `Expression` would
conflict. Python and C# have no such obstacle.

**8. The event `wait(timeout)` implementation is consistently wrong.** Python, C++, C# and
Java all detect timeout by resetting the cached value and comparing it before and after the
wait, rather than checking the wait's return value. All four therefore cannot distinguish a
timeout from an update to the same value, and all four re-apply the timeout per loop
iteration instead of treating it as a total deadline.

**9. Service discovery differs three ways.** Python is hybrid (pre-generated stubs first,
falling back to fully dynamic runtime construction, and it can synthesize enum and exception
types at runtime); Lua is fully dynamic with no codegen at all; C++, C#, Java and cnano are
generated-only, and in Java a service with no generated stub cannot be decoded at all.

**10. Minor signature inconsistencies.** C++ puts `name` first where others lead with the
address; C# takes a `System.Net.IPAddress` with no DNS resolution path, so a user with a
hostname must call `Dns.GetHostAddresses` themselves.

## Server and protocol features no client exposes

**Batched RPCs.** The server support is complete and documented. `RequestContinuation.cs:52-84`
executes all calls in a request sequentially and in order; a failing call does **not** abort
the rest; N results are returned for N calls, each independently a value or an error; and
when one call yields across a game update the `results[i] == null` guard prevents already
completed calls from re-running. Documented at
`doc/src/communication-protocols/messages.rst:59-60`. Every client hard-codes one call in and
reads `results[0]`. This is purely a client-library gap and the highest-value one — batching
is the main lever available for latency-bound control scripts. Tracked as part of #903.

**`AddStream(call, start=true)`.** All four stream-capable clients hard-code `start=false` and
then issue a separate `StartStream`, spending an extra round trip on every stream created.
The eager-start path is unreachable from every client.

**Numeric `service_id` / `procedure_id`.** Only cnano uses them; every other client sends
procedure name strings on every call. Note the documented caveat
(`messages.rst:80-85`): ids may change between server versions, names will not.

**Websockets.** The server has full RFC 6455 support including fragmentation, UTF-8
validation, ping/pong and close codes. No client implements it.

**Procedure metadata at runtime.** `game_scenes`, `return_is_nullable` and `Parameter.nullable`
are consumed only by the offline code generator. A client could pre-validate calls against
the current scene instead of waiting for the server's runtime error. Anyone using
`game_scenes` must know that an **empty list means "available in all scenes"**
(`MessageExtensions.cs:110-117`) — read naively it means exactly the opposite.

**Error fidelity.** `ConnectionResponse.Status` distinguishes `MALFORMED_MESSAGE` / `TIMEOUT` /
`WRONG_TYPE`; every client collapses this to a bare message string. Likewise `Error.service`
and `Error.name` survive only as concatenated message text, never as structured accessors on
the thrown exception.

**Settable server state.** `KRPC.GameScene` and `KRPC.Paused` are both gettable *and settable*
over the wire; scene switching is asynchronous, so a client must poll until the new scene is
reported. These are reachable through the generated stubs, so this is a discoverability and
documentation gap rather than a missing API.

Two related notes that are not client gaps:

- **Multiplexing is serial-only and envelope-only.** `MultiplexedRequest`/`MultiplexedResponse`
  are used solely by the SerialIO transport to disambiguate message kinds on a single
  bidirectional wire. Websockets does not multiplex; it uses a second connection like TCP.
- **Events are never deduplicated**, unlike procedure-call streams — `EventStream.Equals` is
  reference equality. This interacts with the refcounting design (#902).

## Already tracked

Four findings have agreed designs already; this audit adds cross-client evidence but no new
proposal:

| Finding | Issue | Design |
|---|---|---|
| Batched calls (and reverse streams) | [#903](https://github.com/krpc/krpc/issues/903) | [`protocol/reverse-streams.md`](protocol/reverse-streams.md) |
| Duplicate-stream removal breaks other consumers | [#902](https://github.com/krpc/krpc/issues/902) | [`protocol/stream-refcounting.md`](protocol/stream-refcounting.md) |
| Stream invalidation on error | [#877](https://github.com/krpc/krpc/issues/877) | [`protocol/stream-invalidation.md`](protocol/stream-invalidation.md) |
| Stream freezing — C++ only, to be **removed** not ported | [#901](https://github.com/krpc/krpc/issues/901) | [`protocol/stream-freezing-removal.md`](protocol/stream-freezing-removal.md) |

The batching rung of #903 is already scoped there as client API work with no server change
needed, which this audit confirms independently.

This audit's finding that stream freezing exists only in C++ was originally an argument for
porting it to the other clients. The decision went the other way: it is superseded by server-side
function streams (#679) and will be removed rather than propagated.

## cnano bugs, as fixed

Fixed 2026-07-21 in [PR #1002](https://github.com/krpc/krpc/pull/1002).

**Bug 1 — short reads.** One-line offset fix. The test forces a real short read over a pipe by
writing part of the payload, then supplying the remainder from a second thread only once the
reader has blocked; against the unfixed code the second chunk lands at offset 0.

**Bug 2 — the Arduino spin.** Resolved as `readBytes() == 0` → `KRPC_ERROR_EOF`, matching the
POSIX and Windows backends and Python's `socket.error("Connection closed")` on a zero-length
`recv`. An intermediate design added a retry count before giving up, on the theory that a
single timeout might be a slow server rather than a dead one; it was dropped once it became
clear that Arduino's `readBytes` applies its timeout **per byte** (`Stream::timedRead` restarts
the clock for each one), so returning nothing already means total silence for a full timeout
and a slow-but-progressing reply never trips it. A retry count would only have multiplied
`Serial.setTimeout` by a constant — a second knob for something the standard Arduino one
already expresses. This narrows finding 6 above: Arduino now has a working read timeout, POSIX
(`VTIME`) and Windows (`COMMTIMEOUTS`) still do not.

Testing needed scaffolding, since the Arduino backend cannot be built for the host as-is. A
new `//client/cnano:arduino` target compiles `communication.c` with `ARDUINO` defined against a
stand-in `HardwareSerial` with scripted reads, deliberately not linking `krpc_cnano` (whose
host-platform copy defines the same symbols). The stand-in is free of the C++ standard library
because it is included into the backend's own translation unit — and because the real Arduino
has no `<vector>` either. This target needs no server, unlike the rest of the cnano tests.

**Bug 3 — discarded errors.** Opt-in `KRPC_ERROR_MESSAGES`, capturing `service.name:
description` plus the stack trace into a static `KRPC_ERROR_MESSAGE_LENGTH` (default 256) byte
buffer, read with `krpc_get_error_message`. The string fields are copied in fixed 32-byte
chunks as nanopb decodes them, so no buffer scales with the size of the message and a long
stack trace simply truncates. Left undefined, the field callbacks stay null and nanopb skips
the strings exactly as before — the cost really is zero, which is what made opt-in the right
shape rather than a smaller always-on subset.

The test binary is built against a `testonly` library variant with the option enabled, and the
five test-service exception tests assert their messages; the default configuration stays
compiled through `clang_tidy_test`. Worth knowing for anyone extending this: the assertions
compare a prefix, because `TestServer` runs with `VerboseErrors` on and so always sends a stack
trace.

## Java bugs, as fixed

Fixed 2026-07-21 in [PR #1004](https://github.com/krpc/krpc/pull/1004). Each
fix has a test that was confirmed to fail against the unfixed code.

**Bug 4 — `getRate`.** Assign the field in `setRate`, as Python does.

**Bug 5 — method lookup.** Names are now compared by value and each argument against its own
parameter. The trap: arguments arrive boxed through the varargs, so the obvious
`paramTypes[i].isAssignableFrom(args[i].getClass())` rejects **every** primitive parameter —
`float.class.isAssignableFrom(Float.class)` is false — and breaks all existing use. Primitive
parameter types are mapped to their wrappers first, and the existing `floatToString(3.14159f)`
tests are what guard that.

Note there is nothing to test real overload resolution against: kRPC requires unique procedure
names per service, so no generated stub has two same-arity overloads, and none can. The
argument check is therefore tested by rejection (a wrong-typed argument no longer resolves)
rather than by selection. This also means the arity-only matching was never wrong in practice —
the damage was the deferred failure, a `ClassCastException` out of `Encoder.encode` doing
`(double) value` on the wrong box, instead of a clear "method not found".

The non-interned-name half is the one that bit real users: `getName() == methodName` worked
only because both sides are normally interned, so any name assembled at runtime silently failed
to resolve.

**Bug 6 — throwing callbacks.** Each callback runs isolated and anything it throws goes to the
thread's uncaught exception handler. Swallowing was rejected — the update thread has no caller,
so a swallowed exception would hide a broken callback entirely; the default handler still
prints it and a user can install their own. **Python has the identical hole** (bug 20 above),
found while fixing this; there was no reference behavior to copy.

**Bug 7 — lock inversion.** `update` now holds `updateLock` only to find the stream and decode
its value, releasing it before taking the condition, so every path acquires condition then
`updateLock` and never the reverse. Callbacks are snapshotted under the lock, since they now
run outside it.

Reproducing it deterministically is the interesting part: hold a fast-updating stream's
condition, sleep long enough that the update thread is certainly parked on that same condition
while holding `updateLock`, then call `remove()`, which needs `updateLock`. Both threads block
forever and a JUnit `timeout` turns the hang into a failure. Widening the window with the sleep
is what makes it reliable — the natural race is far too narrow to hit on demand, which is
presumably why this survived so long.

**Bug 8 — unknown exception type.** Falls back to an `RPCException` carrying
`service.name: description`. Deliberately *unlike* Python, which raises a bare `RuntimeError`
about the missing type and drops the description (bug 21 above) — the whole complaint here was
that the real error gets masked, so copying Python would have missed the point. The adjacent
case the audit did not mention is fixed too: with no single-argument constructor the old loop
fell through holding the *last* constructor and handed it to `newInstance`, raising an uncaught
`IllegalArgumentException`.

### Second pass: bugs 23, 24, 27, 28, 30, 44

Chosen as the highest impact of the sweep that were also small and safe enough to land against
a release date. What was deliberately left, and why, is as much the point as what was taken:
25 needs request ids to correlate a response with its call, and that is a protocol change; 26
and 41 are behavior changes needing a decision on which side moves; 34 and 36 would start
raising where scripts currently pass and silently succeed. None of those is a patch.

**Bug 23 — the disconnect blindness.** The one with real user impact, and the one whose old
behavior was *pinned by a test*: `test_partial_receive_on_remote_closed_connection` asserted
`b""` at end of file while its sibling `test_receive_on_remote_closed_connection` asserted that
`receive` raises on the identical condition. Two tests locking in deliberately inconsistent
semantics for the same event, which is a good part of why nobody questioned it. Fixed by making
`partial_receive` raise, as `receive` and `send` already did, and the test corrected to match.

**Bug 27 — the regression.** Introduced by this PR's own bug 22 fix and caught only because
the sweep re-read the code afterwards. Recorded here because the same mistake was then found in
the Java bug 7 fix (bug 51) and pre-empted in the C# one: releasing the update lock before
storing a decoded value is not safe on its own, and every client that gets the lock-ordering fix
needs the removed-stream guard with it.

The first attempt at its test proved nothing. Injecting the removal through the error path meant
the value being clobbered was itself a `StreamError`, so the test passed and failed correctly
while never demonstrating the harm. It now clobbers with an ordinary value, which is the real
failure: a removed stream returning stale data indefinitely.

**Bugs 24, 28, 30, 44.** The `sint64` decoder, waking waiters on removal and close, closing from
a callback, and the value/flag publication order. 44 has no test — the window is a single
bytecode, and a test for it would be flaky rather than informative.

## C# bugs, as fixed

Fixed 2026-07-21 in [PR #1005](https://github.com/krpc/krpc/pull/1005).
Bug 16 was left: it is confirmed, but the fix needs a decision rather than a patch.

**Bug 15 — the unguarded indexer.** Confirmed, and fixed as in Java and Python: fall back to an
`RPCException` carrying `service.name: description`. Without it the test fails with exactly the
predicted `KeyNotFoundException`.

**Bug 14 — the undisposed stream manager.** Confirmed, but the obvious fix deadlocks the entire
test suite, which is worth recording because the reason is not visible from the bug report.

Disposing the stream manager stops and joins the update thread. The natural instinct is to do
that *before* closing the sockets, so nothing is pulled out from under the thread. That hangs:
`Connection.ReadMessageData` polls its `stopEvent` only with `WaitOne(0)` *after*
`stream.Read` returns (`Connection.cs:321-323`), so a thread parked in a blocking read never
observes the stop event at all. Closing the socket underneath it is the only thing that
releases it. The join therefore has to come **after** the close — which keeps the socket
closure as the wake mechanism while still making `Dispose` return only once the thread has
actually ended.

The deeper fix is to make that read observe the stop event, at which point the ordering stops
mattering. That is the same shape as the Python disconnect work (bug 23) and belongs with it.

There is no test: the fix changes when the thread ends, not whether it does. The suite is the
regression test in practice — getting the order wrong deadlocked every test in it.

**Bug 16 — the hard cast.** Confirmed. `StreamManager.Update` stores
`connection.GetException(result.Error)` as the stream's value (`:100`) and then passes it to
every callback (`:106-107`), while `Stream<T>.AddCallback` wraps the user callback in
`(object x) => callback((TReturnType)x)` (`Stream.cs:138`). An errored stream therefore casts
an exception to the stream's value type. Left alone because the question it raises — what
should a typed callback receive when its stream errors? — has no obvious answer, and the same
question applies to Python and Java, which pass the error object to callbacks. Worth settling
once for all three rather than patching C# alone.

**Two bugs found while doing this**, recorded above as 49 and 50, both since fixed:

- **49.** Callbacks are invoked bare and `UpdateThread.Main` catches only
  `ObjectDisposedException` and `IOException` — bug 6 (Java) and bug 20 (Python) again. The
  consequence is worse here than in either: on .NET an unhandled exception on a background
  thread terminates the **process**, so a single throwing callback does not just stop the
  streams, it kills the application. Confirmed by running the test against the unfixed code,
  which aborted with a core dump rather than failing. Bug 16 is one way to trigger it, which
  makes the two compound: an errored stream that has a callback takes the process down.
- **50.** The `updateLock` then `condition` ordering in `Update`, against `Event.Wait` setting
  `stream.Value` with the condition held — bug 7 (Java) and bug 22 (Python) again.

Fixing 50 also required carrying over Python bug 27: releasing the update lock before storing
opens a window in which the stream is removed, and the value then overwrites the error marking
it as removed. That guard went in with the fix here rather than being discovered afterwards, as
it was in Python.

**Which then turned up bug 51.** The same reasoning applied backwards found that the Java bug 7
fix had opened that window too and never closed it — introduced and fixed in [PR #1004](https://github.com/krpc/krpc/pull/1004)
before release, so it gets no changelog entry. Java has no seam equivalent to Python's
replaceable module-level decoder, so that fix is untested; the C# guard is likewise covered only
indirectly.

That is now **all four** stream-capable clients carrying both defects. They are not independent
mistakes; they are one design, copied. The "Where the clients already agree" section describes
that design approvingly, which is exactly how a shared bug reads as a shared convention. C# did
get one detail right that Python did not: its value setter assigns the value before the updated
flag (`StreamImpl.cs:60-62`), which is the ordering Python had backwards as bug 44.

## C++ bugs, as fixed

Fixed 2026-07-21 in [PR #1007](https://github.com/krpc/krpc/pull/1007). Bugs 9,
12 and 13 were taken; 10 and 11 deferred, for the reason the sequencing already gives — they are
one piece of work, not two bugs, and that work is giving C++ a teardown story rather than
patching a declaration and a `weak_ptr`.

**Bug 9 — the unchecked `find`.** Confirmed and fixed as in every other client: fall back to an
`RPCError` carrying `service.name: description`. Testable, unusually, because `Client::invoke`
and `add_exception_thrower` are public while `throw_exception` is private: a client that never
constructs `services::TestService` has an empty thrower map, so invoking a throwing procedure by
name on it reaches the missing-key path.

**Bug 12 — the hard read in the callback loop.** Confirmed, and the severity is the C# one, not
the Java or Python one: the unfixed test **core dumped**. An exception leaving a `std::thread`
calls `std::terminate`, so a single throwing callback ends the process.

The fix has a part the other clients did not need. C++ callbacks take the encoded value
(`std::function<void(const std::string&)>`), so there is nowhere to put an exception — an
errored stream has literally nothing to hand a callback. Callbacks are therefore skipped for an
update carrying an error, and the error is still reported to anything reading the stream. This
is bug 16's open question answered by the type system rather than by choice, which is a useful
data point for settling 16: C++ cannot do what Python and Java do, so "pass the error to the
callback" cannot be the cross-client answer.

**Bug 13 — the unlocked map.** Confirmed reachable rather than theoretical: `StreamManager`
turns a stream error into an exception on the update thread through the same map, so any
service constructed while a stream is erroring races it. The mutex is held by `shared_ptr`
because `Client` is copyable and a `std::mutex` is not — the naive `mutable std::mutex` member
deletes the copy constructor and fails to compile against `krpc::connect` returning by value.
The thrower is copied and the lock released before it is called, so the lock is not held while
the exception unwinds. Untested: a data race has no deterministic test.

**Where C++ is better than the others.** It does **not** have bugs 7/22/50, the lock-order
inversion, and not by accident: the stream condition variable has its own mutex, and the update
path never acquires it — `notify_all` does not require it. Nothing takes the two locks in a
cycle. Worth remembering when the other three are reworked.

One thing noticed but not chased: `StreamManager::update` notifies the per-stream condition
without holding that stream's condition mutex (`stream_manager.cpp:82`), while the same file
deliberately does hold the manager-level condition mutex when notifying, with a comment
explaining that a notification fired between a waiter's check and its wait would be lost. The
two are inconsistent, and the per-stream one looks like the unguarded case that comment warns
about. Not verified; it belongs to the hardening pass.

## Lua bugs, as fixed

Fixed 2026-07-21 in [PR #1003](https://github.com/krpc/krpc/pull/1003). All
three were unconfirmed when written down; confirming them first was worth the effort, because
two of the three descriptions turned out to be wrong in ways that changed the fix.

**Bug 17 — `encoder.client_name`.** Confirmed: calling it raises "attempt to perform arithmetic
on field 'CLIENT_NAME_LENGTH' (a nil value)". Nothing calls it, and the connection request sets
the name directly on the protobuf message, so it was removed rather than repaired.

**Bug 18 — the leaked globals.** The description says `ok, value` are both assigned as globals.
Only `ok` is: `value` is the enclosing `for` loop's variable, so it is in scope and the
assignment correctly writes the local. Verified in an interpreter.

This matters, because the obvious fix — `local ok, value = pcall(...)` — introduces a real bug
where there was only namespace pollution. It creates a *new* local `value` scoped to the branch,
while the coerced value is read after the branch ends, so every coerced argument would silently
be sent uncoerced. Only `ok` may be declared.

The test also needed two attempts. Passing a list where a tuple is expected does **not** coerce:
`TupleType` passes `List` as its `lua_type` (`types.lua:370`), so a list already satisfies the
type check and the coercion branch is skipped entirely. The first test therefore passed against
the unfixed code. Passing a plain table where a list is expected does coerce.

**Bug 19 — `self._enum_name`.** Confirmed, and confirmed harmless: `_create_enum_type` ignores
both the service and class names it is given and uses only the values. Fixed by passing
`_class_name`, the field that is actually set. No behavior change, so no test.

## Python bugs, as fixed

Two passes, both in [PR #1008](https://github.com/krpc/krpc/pull/1008). The first pass took bugs 20-22, before the sweep existed; the second took
23, 24, 27, 28, 30 and 44 out of it. What is left of 23-48 is the hardening work in step 4.

### First pass: bugs 20-22

All three are the Java bugs 6, 8 and 7 over again, in the client the audit had been treating
as the reference throughout — which is the finding worth keeping. Python was read closely for
*what it does*, and its behavior taken as the house style, but never audited for defects of its
own. Bug 22 was not even on the list: it was found by reading `update` while fixing bug 20 and
recognizing the shape from Java, then confirmed with the same reproduction.

**Bug 20 — throwing callbacks.** Same shape as bug 6, reported through `threading.excepthook`
rather than Java's uncaught exception handler, so it stays visible by default and an
application can route it. Both stream callbacks and the client-wide update callbacks go
through it.

Its unfixed failure mode is worth recording, because it shows what users actually experience.
The test fails, and then the *rest of the stream suite hangs*: `ServerTestCase` shares one
connection across the class, so once the update thread dies every later test waiting on a
stream waits forever, and the whole run dies on the bazel timeout rather than on an assertion.
That is exactly the user-facing symptom — one bad callback anywhere and every stream on the
connection is silently dead — but as a test failure it is close to undiagnosable. The bug 22
test therefore builds its own connection, so a regression there deadlocks only itself.

**Bug 21 — masked errors.** Returns an `RPCError` carrying `service.name: description`. Note
the old code *raised* where every other path through `_build_error` returns the exception for
the caller to raise, which is part of why it read as deliberate.

**Bug 22 — lock inversion.** Identical fix to Java bug 7: hold `_update_lock` only to find the
streams and decode their values, take each condition after releasing it, read the callbacks
under the lock since they now run outside it. The restructure is slightly larger than Java's
because a single update message carries results for several streams, so the decode pass has to
collect `(stream, value, callbacks)` and dispatch afterwards.

Reproduction is the same recipe: hold a fast-updating stream's condition, sleep long enough
that the update thread is certainly parked on it, then call `remove()`. `RLock` does not help —
the two threads are different, so re-entrancy never applies.

## Python client sweep

Run 2026-07-21, after fixing bugs 20-22. Those three were all Java bugs over again, found only
because the Java work forced the same code to be read twice — which raised the obvious question
of what else was in there. Four passes over the hand-written runtime, split by area (streams and
lifecycle; wire layer; types and dynamic service construction; client core and errors), each
reading its files in full and cross-checking against `protobuf/krpc.proto`, the protocol docs
and the server in `core/src/`. Findings are bugs 23-48 in the table above.

**What this says about the audit's method.** The original sweep asked, for every client, "does
this match Python?". That question cannot produce a finding about Python. Worse, Python's
behavior was written up in "Where the clients already agree" as the house style — so the one
client never checked for correctness is the one whose behavior became the standard the others
are measured against. Two entries in that list are now known to be describing defects:

- "A stream error is stored as the stream's value and rethrown at read time" — true, but in
  Python the store can be clobbered (bug 27) and nothing wakes a waiter (bug 28).
- "One call per request; a single lock serializes each RPC round trip" — the lock serializes but
  does not make the exchange atomic against failure, and nothing correlates a response with its
  request (bug 25).

The parity matrix has the same problem in miniature: Python is credited with "explicit close,
plus context manager", and the context manager turns out to have no test, no documentation and
no example script anywhere. It was scored on existence, not on working.

**Themes, rather than a list of unrelated defects:**

- *Nothing detects a disconnect.* Bug 23 is the root: the client cannot tell "no data yet" from
  "peer gone" anywhere, so it spins instead of failing. This is the same defect as cnano bug 2,
  fixed there first because a microcontroller made it obvious. Bugs 29, 31 and 45 are the same
  blind spot at shutdown.
- *Shutdown is not a designed path.* `close()` can hang forever (29), cannot be called from a
  callback (30), closes the RPC socket while callbacks are still running (31), never wakes
  waiters (28), and has no counterpart for a client that is simply dropped (45).
- *The dynamic and pre-generated paths disagree.* Bugs 26, 32, 41 are all cases where the two
  service-loading modes behave differently, and in each the default (pre-generated) path is the
  broken one. Bug 33 is the leak introduced by the most recent attempt to reconcile them.
- *Legal-but-unusual service definitions kill the whole connection.* Bugs 37-40. Each is a
  `krpc.connect()` that raises instead of one RPC that fails, and each is reachable by a
  third-party service author with no way to discover the constraint.
- *Wrong data returned silently.* Bugs 24, 34, 35, 36, 43 — no exception, just wrong values.
  These are the ones a user cannot debug, and 24 is a straight decoder error.

**Coverage gaps found alongside**, all cases where a test would have caught the above: the
context manager is untested; RPC-only mode (`stream_port=None`) is untested; `Stream.rate`'s
getter is never asserted — the identical gap that hid Java bug 4, where the getter returned zero
forever and the rate test passed anyway; and the `sint64` test cases stop at ~2^40, well below
where bug 24 starts.

## Documentation errors

- **`doc/src/communication-protocols/serialio.rst:100-107`** describes receiving stream updates
  over the serial connection via the `stream_update` field. This is not implemented:
  `SerialIO/StreamServer.cs:14` constructs its base with a `NullServer` and `CreateClient`
  unconditionally throws, so `AddStream` over serial fails with "No stream client is connected
  for this RPC client". Nothing in the SerialIO tree ever writes a `StreamUpdate`. The
  `stream_update` field of `MultiplexedResponse` is dead on the wire. This section should be
  removed or marked as not implemented.
- **`doc/src/cpp/client.rst`** documents a `~Client()` that "closes the connection". No such
  destructor exists (see bug 10); cleanup is implicit via `shared_ptr`.
- **`doc/src/communication-protocols/messages.rst`** documents only `GetStatus`, `GetServices`,
  `AddStream`, `StartStream` and `RemoveStream`. It omits `SetStreamRate`, `AddEvent`,
  `GetClientID`, `GetClientName`, `Clients`, `GameScene`, `CurrentGameScene` and `Paused`.

## Not verified

Whether clients actually omit trailing arguments to exploit the server's
`Parameter.default_value` substitution (`Services.GetArguments`, `Services.cs:319-323`), or
always send them explicitly. The server supports omission and it would save wire bytes on
every call with defaulted parameters. This was outside the audit's rubric and needs checking
before anything is claimed about it.

## Sequencing

This audit was meant to be a quick sweep for cheap fixes before v0.6.0, plus input to client
feature planning for the release after. The Python sweep changed the picture: hardening that
client is now its own piece of work, and it is deliberately **not** being rushed into v0.6.0.

**In v0.6.0**

1. ~~**Bugs 1-3 (cnano)** — smallest, most serious, self-contained.~~ Done ([PR #1002](https://github.com/krpc/krpc/pull/1002)).
2. ~~**Bugs 4-8 (Java)**~~ Done ([PR #1004](https://github.com/krpc/krpc/pull/1004)).
   ~~**Bugs 17-19 (Lua)**~~ Done ([PR #1003](https://github.com/krpc/krpc/pull/1003)).
   ~~**Bugs 14-15 (C#)**~~ Done ([PR #1005](https://github.com/krpc/krpc/pull/1005)).
   ~~**Bugs 9, 12-13 (C++)**~~ Done ([PR #1007](https://github.com/krpc/krpc/pull/1007)).
3. ~~**Bugs 20-22 plus the highest-impact, lowest-risk of 23-48 (Python)**~~ Done ([PR #1008](https://github.com/krpc/krpc/pull/1008)):
   20-22, then 23, 24, 27, 28, 30 and 44. Scoped to what was small and
   safe enough to land against a release date; the rest of 23-48 waits for the hardening pass.

**After v0.6.0**

4. **Harden the Python client** — the rest of bugs 23-48, as its own effort rather than a
   release scramble. Python is the most used client, so it goes before the others. The themes
   above are the natural units of work: disconnect detection first (bug 23 and its family),
   then shutdown, then reconciling the dynamic and pre-generated paths. Several of these want a
   test harness that can drive a fake or half-dead server, which does not exist yet and is
   probably the first thing to build.
5. **Re-run the sweep for every other client, against a hardened Python** — only at that point
   is "does this match Python?" a question worth asking, because only then does matching Python
   mean being correct. Expect this to find in C++, C#, Lua and cnano the same defects the
   Python sweep just found, since the audit's own "where the clients agree" section shows how
   much behavior they share. Bugs 9-19 remain unconfirmed until then.

**Independent tracks, unchanged**

- **Timeouts and error fidelity** — cross-cutting but shallow per client. Overlaps heavily with
  Python bug 23; doing them together is probably cheaper.
- **Batching client APIs** — follows the existing #903 design; no server work.
- **C++ stream leak and the missing `close()`** — bugs 10-11 together, since both are about
  C++ having no deterministic teardown story.
- **Lua streams** — the largest single feature gap, and the one needing real design work
  because the blocker is Lua's concurrency model, not missing plumbing. Worth deciding
  explicitly whether it stays out of scope permanently, in which case the inert
  `add_stream`/`add_event` stubs should be hidden rather than left callable.
